## TL;DR

| Severity | Count |
|---|---|
| CRITICAL | 2 |
| HIGH | 4 |
| MEDIUM | 7 |
| LOW | 5 |
| **Total** | **18** |

**CRITICAL flagged at top per audit rules.** Both are integration mismatches that survived multi-agent + Hermes review because Hermes exercised only the `POST /memories/search-similar` happy path, not the `GET /memories` path that every other MCP tool relies on, and because the retry scheduler was specced but never wired into `index.ts`.

### Top 5 recommended fix order
1. **F-200** ‚Äî MCP `client.searchMemories` reads `result.memories` but `GET /memories` returns `{items}`. Every `memory_search`, `task_status`, `task_list`, `task_complete`, `task_block`, `project_status`, `project_summary`, `commitment_list`, `status_get`, `person_get` returns 0 results. Fix in `packages/omnimind-mcp/src/lib/client.ts:137`.
2. **F-201** ‚Äî `embedding-retry-scheduler` is fully implemented but **never imported or started** in `packages/omnimind-api/src/index.ts`. The WS-2 outbox queue grows unboundedly; pending embeddings are never retried.
3. **F-202** ‚Äî `findNearDuplicate` in `memory.service.ts` does NOT filter by `tenantId`. Cross-tenant dedup merges silently steal memories across tenant boundaries on create.
4. **F-203** ‚Äî HTTP MCP transport fails open when `OMNIMIND_MCP_API_KEY` is unset: the `if (expectedKey && providedKey !== expectedKey)` short-circuits to permit all callers.
5. **F-204** ‚Äî Three retrieval layers (`semantic`, `fulltext`, `trigram`) silently swallow ALL DB errors and return `[]`. A broken pgvector install, an OpenAI 401, a schema-drift error ‚Äî all become "no results" with no signal to the caller.

## CRITICAL Findings

### F-200: MCP `searchMemories` is dead ‚Äî wrong response shape parsed

- **Severity:** CRITICAL
- **Category:** Integration mismatch (schema-vs-code drift between MCP client and API route)
- **Evidence:**
  - Client reads `result.memories ?? []`: `packages/omnimind-mcp/src/lib/client.ts:136-137`
  - API route returns `{items, total, offset, limit}` (from `searchMemories` service): `packages/omnimind-api/src/routes/memories.routes.ts:154-169` calling `memoryService.searchMemories` whose return at `packages/omnimind-api/src/services/memory.service.ts:475` is `return { items, total, offset, limit }`
- **Attack scenario / failure mode:** Every MCP tool that searches via `client.searchMemories` always returns `[]`. That means `memory_search` returns no results; `task_upsert` always takes the "create" branch instead of updating an existing task by title (`packages/omnimind-mcp/src/tools/task.tool.ts:61-78`), creating dup memories on every `task_upsert` call; `task_status`, `task_list`, `task_complete`, `task_block`, `project_status`, `project_summary`, `commitment_list`, `status_get`, `person_get` all return empty.
- **Repro:** From an MCP client, call `memory_write` to create a "Task: foo" memory, then call `task_status({ taskTitle: 'foo' })`. Response: `{ found: false }`. Or call `task_upsert` twice with the same title ‚Äî get two memory rows.
- **Fix:** In `client.ts:137` change `return result.memories ?? [];` to read from `items`. Recommended: change the return-type generic to `{ items: MemoryRecord[] }` and `return result.items ?? [];`. Update API route to also include a `memories` alias if cross-compat is required ‚Äî but a single fix at the client is cleaner.
- **Effort:** ‚â§30min (1 line + type generic update)
- **Deferred?:** No ‚Äî fix immediately.

### F-201: WS-2 embedding retry scheduler is never started

- **Severity:** CRITICAL
- **Category:** Dead code (cron specced + implemented but not wired into bootstrap)
- **Evidence:**
  - Scheduler exports `startEmbeddingRetryScheduler` at `packages/omnimind-api/src/jobs/embedding-retry-scheduler.ts:74`.
  - Grep across `src/`: only `services/memory.service.ts` references the file (in a comment) and `prisma/schema.prisma` references it (in a comment). **Not imported into `index.ts`**: see `packages/omnimind-api/src/index.ts:30-33` ‚Äî imports `cortex-scheduler`, `session-summarizer`, `weekly-digest-scheduler`, `importance-decay-scheduler`, but NOT `embedding-retry-scheduler`.
  - Verified by `grep -n "embeddingRetry\|embedding-retry" packages/omnimind-api/src/index.ts` returning empty.
- **Attack scenario / failure mode:** WS-2 spec'd the outbox pattern: `createMemory` enqueues an outbox row, attempts an embed once, and on failure leaves the row for the cron. With no cron started, any memory whose embedding fails on first attempt is **permanently un-embedded**. Per `POST-IMPLEMENTATION-REVIEW.md:74-86`, this is the active failure mode in prod right now: OpenAI keys auto-revoked ‚Üí every memory written via Hermes/MCP currently has `has_embedding=false` and there is no recovery path.
- **Repro:** Restart `omnimind-api` and grep server logs: no `[embedding-retry] Embedding retry scheduler started` line ever appears (only cortex, session-summarizer, weekly-digest, importance-decay log "started" lines).
- **Fix:** In `packages/omnimind-api/src/index.ts`: add the import alongside the other schedulers (line 33), call `startEmbeddingRetryScheduler()` inside the `app.listen` callback (line 100), and add `stopEmbeddingRetryScheduler()` to the `shutdown` function (line 85).
- **Effort:** ‚â§30min
- **Deferred?:** No ‚Äî same blocker class as the missing-OpenAI-key issue. Fix immediately.

## HIGH Findings

### F-202: `findNearDuplicate` performs cross-tenant dedup

- **Severity:** HIGH
- **Category:** Integration mismatch / cross-tenant data leak
- **Evidence:** `packages/omnimind-api/src/services/memory.service.ts:73-95` ‚Äî the raw-SQL `findNearDuplicate` filters by `user_id`, `embedding IS NOT NULL`, `deleted_at IS NULL`, `status != 'ARCHIVED'`, and `cosine >= threshold`. **No `tenant_id` filter.** Called from `createMemory` at line 147, which then calls `updateMemory` on the dup at line 152.
- **Attack scenario / failure mode:** A `josh-personal` MCP agent writes "Project Alpha kickoff" content. A near-identical memory exists in `josh-business` tenant (sourceWeight 1.0). `findNearDuplicate` returns the business-tenant row. `updateMemory` then runs with the agent's `agentContext.tenantId = josh-personal`, which the contextOverrides block at memory.service.ts:512-518 **rewrites** the business memory's tenantId to josh-personal, also overwriting `agentId` and `sourceWeight`. Effectively a tenant boundary is silently crossed during dedup.
- **Repro:** Seed two memories with same content in different tenants ‚Üí call `createMemory` from tenant A with content matching the tenant-B memory at >0.92 cosine ‚Üí query DB ‚Üí tenant-B row is now stamped tenant A.
- **Fix:** Add `tenantId: string` param to `findNearDuplicate` and include `AND tenant_id = ${tenantId}` in the WHERE. Wire from `createMemory` with `agentContext.tenantId` (skip the dedup search when `agentContext` is undefined, since BoardRoom AI is single-user).
- **Effort:** ‚â§30min
- **Deferred?:** No.

### F-203: HTTP MCP transport fails open when `OMNIMIND_MCP_API_KEY` is unset

- **Severity:** HIGH
- **Category:** Silent fail-open + non-timing-safe compare
- **Evidence:** `packages/omnimind-mcp/src/transports/http.ts:25-31`:
  ```
  const expectedKey = process.env.OMNIMIND_MCP_API_KEY;
  if (expectedKey && providedKey !== expectedKey) { ... 401 ... }
  ```
- **Attack scenario / failure mode:**
  1. Unset env-var: if `OMNIMIND_MCP_API_KEY` is absent the truthiness short-circuit lets every request through with no auth at all. The HTTP listener exposes all 15 MCP tools on port 3334 to anyone who can reach the port.
  2. Even when set: `providedKey !== expectedKey` is a plain string compare, not timing-safe.
- **Repro:** Run `node packages/omnimind-mcp/dist/index.js http` without setting `OMNIMIND_MCP_API_KEY`. `curl localhost:3334 -X POST -d '{"jsonrpc":"2.0","method":"tools/list","id":1}'` succeeds.
- **Fix:** Refuse to start in HTTP mode when `OMNIMIND_MCP_API_KEY` is unset. Use `timingSafeEqual` for the compare.
- **Effort:** ‚â§30min
- **Deferred?:** No ‚Äî single-user local-dev exposure is real if Josh ever runs HTTP mode without setting the env var.

### F-204: Three retrieval layers swallow all errors ‚Üí silent zero-result responses

- **Severity:** HIGH
- **Category:** Silent fail-open
- **Evidence:**
  - `packages/omnimind-api/src/retrieval/semantic-search.ts:81-84`: `} catch (err) { return []; }`
  - `packages/omnimind-api/src/retrieval/fulltext-search.ts:140-143`: `} catch { return []; }`
  - `packages/omnimind-api/src/retrieval/trigram-search.ts:94-97`: `} catch { return []; }`
- **Attack scenario / failure mode:** pgvector extension not installed, `tenant_id` column missing after a partial migration, OpenAI 401 from `generateEmbeddingWithRetry`, malformed Postgres response ‚Äî all become "no semantic / FTS / trigram results" with no log. F-107 (the prior `as any` cast) was THE example of this category but the swallow is still present in all three files. The ranker silently degrades to whatever single layer is working (often only `structured-filter`, which is the slowest, dumbest layer).
- **Repro:** Set `DATABASE_URL` to a postgres without `vector` extension, call `POST /context/assemble` ‚Äî get only structured results, no error.
- **Fix:** Log the error with `logger.error('[layer] retrieval failed', { error })` AND return `[]`. The retrieval system should tolerate one layer failing, but it should NOT do so silently.
- **Effort:** ‚â§30min
- **Deferred?:** No.

### F-205: Several MCP tools never normalize `domain` ‚Üí ministry refusal still bypassable in non-memory paths

- **Severity:** HIGH
- **Category:** Integration mismatch (WS-6 F-101 fix incomplete)
- **Evidence:**
  - `packages/omnimind-mcp/src/tools/decision.tool.ts:11` ‚Äî `domain: z.string().default('business')` (no transform, no normalize). The decision tool then writes via `client.createMemory({domain: input.domain, ...})` at line 29. Server-side `memory.service.ts:125` does normalize, so the DB row gets the lowercased value ‚Äî BUT the MCP-side audit redaction (`packages/omnimind-mcp/src/lib/audit.ts:writeAuditLog`) records the raw input. A caller passing `domain: 'Ministry'` via `decision_log` would be refused by the server but the un-redacted ministry content lands in the audit log.
  - `packages/omnimind-mcp/src/tools/task.tool.ts`, `commitment.tool.ts` ‚Äî neither has a domain input field; they hardcode `'business'` ‚Äî fine for those.
  - The `decisionLogTool` test omitted: server gates ministry but redaction occurs only in `memory.tool.ts:redactForAudit`.
- **Attack scenario / failure mode:** Hostile or misconfigured MCP client calls `decision_log({ domain: 'Ministry', content: '<sensitive>' })`. Server-side gate refuses with 503 MINISTRY_DEFERRED. But the audit log captured `inputJson` already contains the full content. F-101 closed the WRITE leak; the AUDIT leak is still open.
- **Repro:** Call `decision_log({ title: 'X', content: 'pastoral note', domain: 'Ministry' })` ‚Üí check `mcp_audit_logs.input_json` ‚Üí contents present in cleartext.
- **Fix:** Reuse the `DomainSchema` from `memory.tool.ts` (or extract to a shared normalize) and apply to all tools that accept a `domain` input. Add ministry-redaction to `withAudit` itself if `inputJson.domain === 'ministry'`.
- **Effort:** ‚â§2hrs
- **Deferred?:** No.

## MEDIUM Findings

### F-206: `GET /mcp/audit` has no default tenant scope

- **Severity:** MEDIUM
- **Category:** Integration mismatch (admin endpoint, tenant filter optional)
- **Evidence:** `packages/omnimind-api/src/routes/mcp.routes.ts:53-68` ‚Äî `tenantId` is read from `req.query.tenantId` and applied only if present. No fallback to `req.agentContext?.tenantId`. Compare to `admin.routes.ts:94-127` (`/admin/audit`) which DOES resolve from context.
- **Failure mode:** An MCP agent with `OMNIMIND_API_KEY` (i.e. every agent ‚Äî that's the single shared key) hitting `GET /mcp/audit` sees the audit entries of every tenant. F-102 closed this for `/admin`; the duplicate path under `/mcp/audit` remained open.
- **Fix:** Mirror the `resolveTenantFilter` pattern from `admin.routes.ts` for `/mcp/audit`. Or consider that `/mcp/audit` GET is unused (it's not called by any client in the codebase) and remove the GET route ‚Äî only POST is needed.
- **Effort:** ‚â§30min
- **Deferred?:** No.

### F-207: `rateLimiter` and `agentRateLimiter` both skip on missing header ‚Üí bypass via header-omission

- **Severity:** MEDIUM
- **Category:** Silent fail-open
- **Evidence:**
  - `packages/omnimind-api/src/middleware/rate-limiter.ts:20-23`: `if (!userId) { next(); return; }` ‚Äî drops rate limiting entirely if `x-user-id` missing.
  - `packages/omnimind-api/src/middleware/agent-rate-limiter.ts:38-42`: `if (!agentId) { next(); return; }` ‚Äî same for `x-agent-id`. (This is WS-6 F-105, but escalated: F-105 was "mitigated by global per-IP limiter" ‚Äî but the global limiter ALSO bypasses on missing `x-user-id`.)
- **Failure mode:** A holder of `OMNIMIND_API_KEY` omitting both headers gets unlimited request rate. There is no per-IP fallback. Both rate limiters fail-open together.
- **Repro:** Hit `/memories` endpoint with only `x-api-key` set, no `x-user-id` or `x-agent-id`. The user-id check at the route returns 400 before count is incremented, so this doesn't actually achieve DOS ‚Äî but routes that don't require `x-user-id` (e.g. `/admin/stats`) do allow unlimited requests.
- **Fix:** Fall back to `req.ip` for rate-limit key when both headers absent. Or refuse the request entirely when neither header is present (production callers always have at least one).
- **Effort:** ‚â§2hrs
- **Deferred?:** Escalate F-105 ‚Äî solo-mode mitigation is overstated.

### F-208: `keygen` SQL fallback uses wrong table name

- **Severity:** MEDIUM
- **Category:** Schema vs. code drift
- **Evidence:** `packages/omnimind-mcp/src/keygen.ts:46-48`: `INSERT INTO "Agent" (...)` and `ON CONFLICT (name)`. The Prisma model maps to `@@map("agents")` (lowercase) at `prisma/schema.prisma:787`.
- **Failure mode:** When the API registration path fails (e.g. dev DB down) the operator copies the fallback SQL into psql and gets `ERROR: relation "Agent" does not exist`. Onboarding gets blocked.
- **Repro:** Stop omnimind-api, run `keygen` against any postgres ‚Äî SQL output will fail to execute.
- **Fix:** Change `"Agent"` to `"agents"` and `(id, name, api_key_hash, ...)` columns are correct as-is. Verify `gen_random_uuid()` matches `cuid()` semantics ‚Äî actually Prisma uses `cuid()` for `id`, so the manual SQL also drifts on the ID format.
- **Effort:** ‚â§30min
- **Deferred?:** No.

### F-209: `setInterval`s in rate limiters leak on shutdown

- **Severity:** MEDIUM
- **Category:** Resource leak
- **Evidence:**
  - `packages/omnimind-api/src/middleware/rate-limiter.ts:12-17`: `setInterval` at module load, never cleared.
  - `packages/omnimind-api/src/middleware/agent-rate-limiter.ts:18-23`: same pattern.
- **Failure mode:** On `SIGTERM` (`index.ts:80-88`), Prisma disconnects, schedulers stop, but the two cleanup intervals keep running. The process waits for them before exiting cleanly. Not a leak in long-running production (process dies anyway), but it does prevent graceful shutdown ‚Äî Node will eventually hard-exit. Also harms test cleanup (vitest hangs).
- **Fix:** Export `stopRateLimiter()` / `stopAgentRateLimiter()` that `clearInterval(handle)`, call from `shutdown()`.
- **Effort:** ‚â§30min
- **Deferred?:** Yes if not affecting tests. Trigger to revisit: vitest hang report or production zombie processes.

### F-210: `/admin/contradictions` has no tenant filter

- **Severity:** MEDIUM
- **Category:** Integration mismatch (admin cross-tenant)
- **Evidence:** `packages/omnimind-api/src/routes/admin.routes.ts:186-203` ‚Äî query `where: { resolvedAt: null }` with no `tenantId` filter and no `userId` filter. Unlike `/admin/stats`, `/admin/agents`, `/admin/audit`, `/admin/memories`, `/admin/duplicates` (all tenant-scoped via `resolveTenantFilter` post-F-102), this one was missed.
- **Failure mode:** Same shape as F-102 prior to its fix ‚Äî any caller with `OMNIMIND_API_KEY` sees every tenant's contradiction alerts (which can include semi-sensitive titles).
- **Fix:** Apply `resolveTenantFilter(req)` and add `tenantId` to the where clause when not includeAllTenants. ContradictionAlert model already has tenant info on the linked memories ‚Äî may need a join via the memoryId FK to filter.
- **Effort:** ‚â§2hrs
- **Deferred?:** No.

### F-211: `setInterval` in `agent-rate-limiter.ts` runs every 10min in test envs too

- **Severity:** MEDIUM (test hygiene)
- **Category:** Resource leak / dead code in test
- **Evidence:** `packages/omnimind-api/src/middleware/agent-rate-limiter.ts:18-23` ‚Äî module-level `setInterval` fires even when imported by tests.
- **Failure mode:** vitest test runs that import the middleware end up with dangling timers; combined with the absence of `unref()`, vitest cannot exit cleanly without `--forceExit`. Likely contributes to the 53 pre-existing test failures noted in POST-IMPLEMENTATION-REVIEW.md (line 51).
- **Fix:** Add `.unref()` to the interval handle so Node can exit even if it's pending. OR gate the `setInterval` call behind `if (process.env.NODE_ENV !== 'test')`.
- **Effort:** ‚â§30min
- **Deferred?:** Yes (test infra). Revisit alongside test-debt cleanup.

### F-212: Ministry refusals at MCP layer never get audit-logged

- **Severity:** MEDIUM
- **Category:** Dead code / observability gap
- **Evidence:** `packages/omnimind-mcp/src/tools/memory.tool.ts:60-68` ‚Äî the ministry refusal returns early BEFORE `withAudit` is called. No audit row is created for refused ministry writes.
- **Failure mode:** If an agent is misconfigured (or hostile) and repeatedly attempts ministry writes, there's no audit trail of the attempts. F-106 covered "audit succeeds-then-fails-silently"; this is "audit never fires at all for a class of refusals." Forensics on a ministry-attempt incident would find nothing in `mcp_audit_logs`.
- **Fix:** Move the ministry check INSIDE the `withAudit` callback, OR explicitly call `writeAuditLog` before the early return with `errorMessage: 'MINISTRY_DEFERRED'`.
- **Effort:** ‚â§30min
- **Deferred?:** No.

### F-217: `agent-context.ts` `parseSourceWeight` accepts NaN-looking inputs as 0

- **Severity:** LOW
- **Category:** Type narrowing escape (`Number()` of garbage returns NaN, then `Math.max(0, ...)` returns 0)
- **Evidence:** `packages/omnimind-api/src/middleware/agent-context.ts:44-53` ‚Äî `Number("abc")` is `NaN`; the `if (!Number.isFinite(n)) return null;` correctly handles that. But values like `"0xFF"` or `"1e10"` parse as numbers and are clamped ‚Äî `1e10 ‚Üí 2` (max-clamp). Functionally fine but a hostile header value that crashes nothing but sets sourceWeight=2 might be unexpected.
- **Failure mode:** Hermes-trust agent sets `x-source-weight: 1e308` ‚Üí server clamps to 2 ‚Üí that agent ranks above all others as a sourceWeight tiebreaker. Mild ranking abuse.
- **Fix:** Stricter regex check before `Number()` ‚Äî `/^\d+(\.\d+)?$/`.
- **Effort:** ‚â§30min
- **Deferred?:** Yes.

### F-218: `decrypt()` swallows GCM auth-tag failure for non-colon ciphertexts only (F-109 scope check)

- **Severity:** REVIEW NEEDED
- **Category:** Crypto fail-soft (potentially F-109 wider than originally categorized)
- **Evidence:** `packages/omnimind-api/src/lib/crypto.ts:23-35`. WS-6 F-109 said this returns ciphertext on auth-tag mismatch. Re-checking: the function returns `encoded` (the original colon-separated input) on three paths: (1) ENCRYPTION_KEY unset (line 24), (2) no `:` in input (line 25), (3) split parts missing (line 27), (4) decryption throws (line 33). For path (4), `encoded` is `iv:tag:ciphertext` ‚Äî that's the raw ciphertext concatenation, NOT the underlying bytes. So a caller treating the return as plaintext would see "1234:5678:abcd‚Ä¶" ‚Äî recognizable as gibberish, not silent tampering.
- **Question:** Is this still a fail-soft vulnerability per F-109? The audit categorized it as MEDIUM and deferred. My reading is the function returns OBVIOUSLY non-plaintext data, so a caller can detect tamper visually. But a programmatic caller treating any non-null return as success is still vulnerable. Recommend honoring F-109's verdict.
- **Status:** Confirms F-109. Not a new finding.
